## Contexto

Tu registro confirma dos cosas:
1. **El flujo press→grabar / release→transcribir ya existe** (`hotkey.rs:run_rdev_listener` distingue `KeyPress`/`KeyRelease`; `pipeline.rs:203-243` implementa on_press/on_release). Transcribió correctamente "¡Muchas gracias!" y "¡Aquí está el video!".
2. **La tecla NO está bloqueada** y hay **rebote fantasma**: mantener F8 genera ciclos Recording→Processing→Recording cada ~160ms (búferes `AudioTooShort` + alucinación final "Pu-p-p-p-p..."). Causa: `rdev::listen` usa `RegisterRawInputDevices` (observación pasiva — la tecla llega a otras apps) y `on_release` no tiene guarda de reentrada.

## Solución: 3 cambios en 3 archivos

### 1. Habilitar `unstable_grab` en `crates/oido-platform/Cargo.toml`
- `rdev.workspace = true` → `rdev = { workspace = true, features = ["unstable_grab"] }`
- Sin esto, `rdev::grab` ni compila. Es el único cambio de dependencias.

### 2. Cambiar `listen`→`grab` con supresión en `crates/oido-platform/src/hotkey.rs`
- Renombrar `run_rdev_listener` → `run_rdev_grab`. Cambiar `rdev::listen(callback)` por `rdev::grab(callback)`.
- **Callback ahora devuelve `Option<Event>`** en vez de `()`:
  - `KeyPress` del target + mods matchean → `press_tx.send(())` + **`return None`** (suprime la tecla).
  - `KeyRelease` del target → `release_tx.send(())` + **`return None`** (suprime).
  - Modificadores (Ctrl/Shift/Alt/Meta) y teclas no-target → **`Some(event)`** (pasan a la app con foco). Así evitamos modificadores "pegados".
  - Todo lo demás (mouse, etc.) → **`Some(event)`** por defecto al final del match. **CRÍTICO**: el hook de grab instala también un mouse hook; si no pasamos `Some(event)` por defecto, el mouse se bloquea.
- La lógica de matching de mods/tecla y la ventana `MODIFIER_WINDOW_MS` se conservan intactas.
- Mapear `rdev::GrabError` → `PlatformError::Hotkey` (cambia el tipo de error de `listen` a `grab`).
- Actualizar el doc-comment del módulo: ya no es "rdev::listen pasivo", es "rdev::grab con supresión". Documentar el requerimiento nuevo en Linux (evdev requiere grupo `input`/`plugdev`, pero ahora soporta Wayland — mejora).
- **`key_grab.rs` NO se toca**: sigue con `rdev::listen` (observación pasiva es correcta para capturar la tecla interactivamente en `--set-hotkey`; además nunca corre a la vez que el pipeline).

### 3. Máquina anti-reentrada en `crates/oido-core/src/pipeline.rs`
- `on_press` (línea 203): ya tiene `if !s.recording` → no cambia.
- `on_release` (línea 217): **añadir guarda `if !s.recording { return; }` al inicio**. Hoy siempre setea `recording=false` y hace snapshot sin importar el estado, lo que genera los `AudioTooShort` espurios cuando un `KeyRelease` fantasma llega en Idle.
- Resultado: la máquina de estados queda idempotente:
  - Idle + press → Recording
  - Recording + press → no-op (ya grabando)
  - Recording + release → Processing (snapshot + enqueue)
  - Idle + release → **no-op** (elimina el rebote)
- Actualizar el doc-comment del módulo para documentar la anti-reentrada.

## Qué NO cambia
- El trait `Hotkey` (sigue `on_press`/`on_release` `Box<dyn Fn() + Send>`). La supresión vive dentro de `RdevHotkey`.
- El flujo del pipeline (capture→audio-rx→buffer, stt_tx→worker→transcribe→filter→inject).
- Canales (crossbeam), mutexes (parking_lot), STT (whisper-rs). Cumple R1/R2/R3.
- `main.rs`, `key_grab.rs`, `oido-config`, `oido-stt`.
