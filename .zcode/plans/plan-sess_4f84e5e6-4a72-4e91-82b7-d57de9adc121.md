# Plan: Modo "Chunked" — fragmentación de audio largo con transcripción palabra-completa

## Resumen de decisiones (confirmadas contigo)
- **Estrategia de corte:** Timestamps por token de whisper + carryover. La transcripción de cada bloque devuelve dónde termina la última palabra completa; el audio sobrante se arrastra como prefijo del siguiente bloque. Sin overlap, sin retranscripción.
- **Inyección:** Pegado incremental con `inject()` (Ctrl+V) por cada bloque transcrito.
- **Modo:** Nueva variante `Chunked` en `SttMode` (tercera opción junto a Batch/Streaming). No rompe nada existente.
- **Bloque:** 5s por defecto (80.000 samples @ 16kHz).

---

## Arquitectura del pipeline Chunked

```
cpal → [audio_rx bounded 1024] → consumer thread ─┐
                                                   │ (cuando buffer ≥ 5s)
                                                   ▼
                                         [chunk_tx bounded 4]
                                                   │
                                                   ▼
                                           worker STT (1 hilo)
                                          transcribe_timed()
                                          injector.inject(text)
                                                   │
                                          carryover = chunk[cut..]
                                                   │
                                                   ▼
                                        [carryover_tx bounded 1]
                                                   │
                            ◄──────────────────── ┘ (se prepone al próximo chunk)
```

**¿Por qué 1 worker y no 2?** El carryover impone una dependencia secuencial: el bloque N+1 necesita el carryover del bloque N. Con 2 workers el ordenamiento de salidas se vuelve complejo sin aportar throughput real (whisper no paraleliza bien con oversubscription). El "paralelismo" real viene del **pipelining**: mientras el worker transcribe el bloque N, el consumer sigue acumulando el bloque N+1. Con GPU, la transcripción de 5s tarda ~1-2s, así que el sistema procesa más rápido de lo que se graba (tiempo real).

**Manejo de backpressure:** si el worker tarda más de 5s (CPU lenta), el consumer NO corta y sigue acumulando audio hasta que el carryover del bloque anterior esté disponible. Esto evita perder palabras; el bloque simplemente crece más allá de 5s temporalmente. Hay un cap de seguridad de 30s.

### Capa 2 — `oido-core` (nuevo `chunked_pipeline.rs`)

**Archivo: `crates/oido-core/src/chunked_pipeline.rs` (NUEVO)**

Estructuras:
```rust
pub struct ChunkedPipelineConfig {
    pub capture: Box<dyn CaptureSource>,
    pub transcriber: Arc<dyn Transcriber>,
    pub injector: Arc<dyn Injector>,
    pub hotkey: Box<dyn Hotkey>,
    pub hotkey_binding: String,
    pub chunk_duration_secs: f32,  // default 5.0
}

pub struct ChunkedPipeline { /* campos análogos a Pipeline */ }
```

Componentes internos:
- `BufferState { samples: Vec<f32>, recording: bool }` tras `Arc<parking_lot::Mutex<_>>` (regla R3: único mutex del crate, acotado).
- Canal audio `bounded(1024)` (cpal → consumer).
- Canal chunk `bounded(4)` (consumer → worker) — tipo `Vec<f32>`.
- Canal carryover `bounded(1)` (worker → consumer) — tipo `Vec<f32>`.
- Canal event `unbounded()` (estado → observer).

**Constantes:**
- `CHUNK_SIZE = chunk_duration_secs * 16_000` (default 80.000).
- `MAX_BUFFER_SAMPLES = 30 * 16_000` (cap anti-OOM; si se excede, se descarta audio viejo).

**Consumer thread (`oido-audio-chunked`):**
- `select!` entre `audio_rx` y `carryover_rx`:
  - Al recibir frame: si `recording`, resamplea y extiende `samples`.
  - Tras extender, verifica si `samples.len() >= CHUNK_SIZE`. Si es así:
    1. `carryover_rx.try_recv()` — si hay carryover del bloque anterior, lo prepone al inicio del buffer.
    2. Drena `CHUNK_SIZE` muestras del buffer → `chunk_tx.send(chunk)`.
  - Si el buffer excede `MAX_BUFFER_SAMPLES` y no hay carryover disponible (worker atascado), trunca descartando lo más viejo con un `warn`.

**Worker thread (`oido-stt-chunked`, 1 solo):**
- `chunk_rx.recv()` → bloque.
- `transcriber.transcribe_timed(&chunk, CHUNK_SIZE)` → `WordTimings`.
- `phrase_filter::filter(&timings.text)` — reusa el anti-alucinación existente.
- `injector.inject(&timings.text)` — Ctrl+V incremental.
- `carryover = chunk[timings.last_word_end_sample..].to_vec()`.
- Si `!carryover.is_empty()`: `carryover_tx.send(carryover)` (si el canal está lleno, loguea warn y descarta — no bloquea).
- Emite `Idle` tras cada bloque procesado (el tray parpadea Processing→Idle por bloque).

**Callbacks hotkey:**
- `on_press`: `recording = true`, `samples.clear()`. Igual que batch.
- `on_release`: `recording = false`, toma el resto del buffer (lo que no llegó a 5s), lo manda como último chunk por `chunk_tx`. El worker lo transcribe completo (con `max_samples = chunk.len()`, sin recorte) y el carryover final se ignora. Mismo guard anti-reentrada que batch.

**`shutdown()` / `Drop`:** análogo a `Pipeline` — para captura, desregistra hotkey, dropea canales.

**`lib.rs` de oido-core:** exportar `ChunkedPipeline` y `ChunkedPipelineConfig`.

**Tests:** test e2e con transcriber mock + injector mock (reusando el patrón de `pipeline_e2e.rs`) que verifique:
- Un dictado de "12s" produce ~2-3 inyecciones.
- El carryover se propaga entre bloques.
- on_release procesa el resto.

## Reglas del proyecto respetadas
- **R1:** todo el paso de mensajes es `crossbeam::channel` (audio, chunk, carryover, event). Sin `Arc<Mutex<T>>` fuera del `BufferState` acotado.
- **R2:** el nuevo `transcribe_timed` usa `token_data()` que es safe Rust en whisper-rs 0.16. Sin nuevo `unsafe`.
- **R3:** el único mutex de oido-core es `BufferState` (igual que `Pipeline` y `StreamingPipeline`).
- **Errores:** `thiserror` con `SttError` reutilizado; binario usa `anyhow::Context`.
- **Observabilidad:** `tracing` spans (`chunked.cut`, `chunked.infer`, `chunked.inject`) con medición de latencia por bloque (reusa el patrón `LatencyReport` de `pipeline.rs`).
