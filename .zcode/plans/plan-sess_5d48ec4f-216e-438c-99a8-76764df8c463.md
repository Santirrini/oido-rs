## Mantenimiento profesional de oido-2.0: arranque rápido + menú modular

### Contexto

El usuario pidió "limpieza" del código para que "ande más rápido" sin romper nada. La exploración reveló que el `main.rs` actual bloquea **30-40s en el primer inicio** por:

1. **Carga síncrona del modelo whisper** (`main.rs:418-493`) — `load_model` mmap+parsea 140MB–1.5GB en el hilo principal.
2. **Warm-up de 30s de silencio** (`whisper_cpp.rs:352-372` y `streaming.rs:109-121`) — corre una inferencia completa sobre silencio.
3. **Subproceso bloqueante para VAD** (`main.rs:94-151`) — si el modelo VAD falta, lanza `powershell`/`bash` síncronamente.
4. **Sin menú modular** — `tray.rs::build_menu` (250 líneas) y `spawn_event_forwarder` están acoplados; añadir un item requiere tocar 4 archivos.
5. **Sin wizard de primer arranque** — no existe detección de "primer launch" ni onboarding (previsto en `plans/PLAN.md` Fase 3).

El usuario eligió:
- **Estrategia arranque:** lo más profesional → carga diferida (lazy load) con estado "Cargando…" en el tray.
- **Menú:** minimalista, robusto, profesional → refactor interno en módulos + trait `MenuSection` para registro.
- **Restricciones:** tests e2e pasando, comportamiento del hotkey idéntico, build limpio en Win/Linux/Mac, compat de `config.json`.
- **Orden:** Diagnóstico → Lazy load → Menú modular (3 commits separados).

#### **Fase 2 — Lazy load del modelo whisper (mantenimiento profesional)**

**Objetivo:** pasar de ~30-40s de arranque a <1s. La carga del modelo + warm-up se difiere a la **primera pulsación del hotkey** (o a un comando de menú "Pausar/Reanudar" → "Precalentar").

**Cambios en `crates/oido-stt/src/lib.rs`:**

- Añadir variante al trait `Transcriber`:
  ```rust
  /// Indica si el modelo está cargado en memoria. Default: true.
  fn is_loaded(&self) -> bool { true }
  ```
  (Default `true` para que backends que cargan eagerly no necesiten cambios.)

- En `WhisperCpp` y `LocalAgreementStreamer`, añadir flag interno `loaded: AtomicBool` (o `parking_lot::Mutex<bool>` dentro de `whisper_cpp.rs`/`streaming.rs`). Tras `load_model` se setea `true`. Implementar `is_loaded`.

**Cambios en `crates/oido/src/main.rs`:**

- **Antes** (líneas 404-495): se construyen `transcriber`/`streamer` y se llama `load_model` + `warm_up` en el startup.
- **Después:**
  - Crear el `WhisperCpp`/`LocalAgreementStreamer` con `loaded = false` (no llamar a `load_model`).
  - El `transcriber`/`streamer` se construye con sus parámetros (idioma, gpu, n_threads) pero sin tocar el archivo.
  - `warm_up` se elimina del startup.
  - `Pipeline::new` / `StreamingPipeline::new` ahora reciben el backend sin modelo cargado.

- En el callback `on_press` del hotkey (`oido-platform/src/hotkey.rs` integración + `main.rs:544-601`):
  - Antes de propagar el evento al pipeline, verificar `is_loaded()`. Si no:
    - `tray.set_state(TrayState::Loading, theme)` (nueva variante, ver Fase 3).
    - Disparar el warm-up + load en un hilo aparte (no bloquea el callback de hotkey, que cumple regla R1: usar canales).
    - Mientras carga, el callback **encola la pulsación** en el canal `start_tx` (1 slot) o un nuevo `pending_press: Mutex<bool>` interno al pipeline.
    - Cuando termina la carga, el worker emite `PipelineEvent::Ready` y el pipeline procesa la pulsación pendiente.

- **Estados del pipeline** (en `oido-core/src/pipeline.rs` y `streaming_pipeline.rs`):
  - Añadir variante `Loading` a `TrayState` (en `oido-platform/src/traits.rs:69-76`).
  - Nuevo `PipelineEvent::Ready` para señalar fin de carga lazy.

- **Caso de fallo:** si `load_model` falla (modelo no existe), emitir `TrayState::Error` + log. La pulsación se descarta; el usuario puede descargar el modelo y volver a intentar.

- **Indicador visual:** `icon::render_state` (`oido-platform/src/icon.rs`) gana un branch para `TrayState::Loading` (gris con barra animada imposible sin tiempo → color gris neutro).

**Compatibilidad:**
- `Config::default()` no cambia.
- `oido-config` no se toca.
- Tests e2e: los mocks ya tienen `is_loaded: true` por default → pasan sin cambios. Si algún test rompe, ajustar mock para implementar `is_loaded`.

**Verificación:**
- `cargo build --workspace` limpio en Win/Linux/Mac.
- `cargo test --workspace` → 27+ tests pasando.
- Manual: ejecutar bin, ver tray en <1s, pulsar F8 → carga + primer dictado ocurre; segunda pulsación es inmediata.

#### **Fase 3 — Menú modular (minimalista, robusto, profesional)**

**Objetivo:** desacoplar la construcción del menú nativo en módulos pequeños, añadir un trait `MenuSection` para registrar secciones sin tocar 4 archivos, e implementar correctamente los placeholders `TogglePause` y `CheckUpdates` (moverlos a Fase 4+ — fuera de alcance de este refactor, solo dejar la arquitectura lista).

**Cambios en `crates/oido-platform/src/`:**

- **Nuevo archivo `tray/sections.rs`** — trait `MenuSection`:
  ```rust
  pub trait MenuSection: Send + Sync + 'static {
      fn id(&self) -> &'static str;
      fn label(&self) -> &str;
      fn build(&self) -> Section;
  }
  pub enum Section {
      Item(MenuItemSpec),
      Submenu { label: String, items: Vec<MenuItemSpec> },
      Separator,
  }
  pub struct MenuItemSpec {
      pub id: String,
      pub label: String,
      pub enabled: bool,
  }
  ```
  `MenuItemSpec` es **agnóstico del backend** (no usa `tray_icon::menu::MenuId`). La traducción a `tray_icon` ocurre en `build_menu`.

- **Nuevo archivo `tray/theme_section.rs`** — `ThemeSection` implementa `MenuSection` con label "Tema" y 3 items (Dark/Light/Sistema).
- **Nuevo archivo `tray/mode_section.rs`** — `ModeSection` implementa `MenuSection` con label "Modo de dictado" y 2 items.
- **Nuevo archivo `tray/hotkey_section.rs`** — `HotkeySection` con item "Cambiar hotkey…".
- **Nuevo archivo `tray/utilities_section.rs`** — "Abrir carpeta de modelos", "Buscar actualizaciones" (placeholder sigue).
- **`tray.rs`** queda solo con: `PlatformTray` (el wrapper), `state_tooltip`, y la función `build_menu` reducida a ~50 líneas que itera sobre `Vec<Box<dyn MenuSection>>` y emite el árbol.

- **Cambio en `tray.rs`:** `WindowsTray::new` y `MacTray::new` aceptan `sections: Vec<Box<dyn MenuSection>>` (o un builder). Se añade `PlatformTray::with_sections(sections)`. La impl Linux (stub) acepta el mismo parámetro y lo loguea (manteniendo compat Win/Linux/Mac).

- **Cambio en `spawn_event_forwarder`:** en vez de un `MenuIdMap` con 9 campos hardcoded, recibe `Vec<(MenuId, MenuAction)>` o un `HashMap<MenuId, MenuAction>`. La función `build_menu` rellena el map durante la construcción.

- **`MenuAction`** (`traits.rs:80-88`) no cambia de variantes. La acción de "toggle pause" se mantiene en el enum (la UI ya está lista, solo faltaba conectar — pero como el usuario quiere conservador, se queda como placeholder, sin tocar su semántica).

**Cambios en `crates/oido/src/main.rs`:**

- En `main()` (líneas 362-364), construir el tray con:
  ```rust
  let sections: Vec<Box<dyn MenuSection>> = vec![
      Box::new(HotkeySection),
      Box::new(ThemeSection),
      Box::new(ModeSection),
      Box::new(UtilitiesSection),
  ];
  let mut tray = PlatformTray::with_sections(sections).ok();
  ```
- Eliminar la rama `MenuAction::TogglePause` del match del listener (líneas 526-528) que es placeholder — **mejor mantenerla** para no romper el `MenuAction` enum (regla: no romper tests).

**Compatibilidad:**
- `MenuAction` enum sin cambios → tests e2e (que mockean el tray) no se rompen.
- Build cfg(Windows/Unix/Linux) preservado.
- `oido-config` no se toca.
- `Cargo.toml` de `oido-platform`: ya no necesita cambios (los nuevos archivos son módulos internos).

**Verificación:**
- `cargo build --workspace` limpio.
- `cargo test --workspace` → 27+ tests pasando.
- Manual: el menú nativo se ve idéntico, los items funcionan igual.

### Garantías

- **Tests e2e pasando:** el `MenuAction` no cambia; el trait `Transcriber` gana método con default → mocks existentes compilan.

- **Compatibilidad `config.json`:** el schema `Config` no se toca; los tests `backward_compat_*` siguen pasando.
